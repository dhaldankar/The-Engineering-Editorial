# Requirements Document

## Introduction

The `packages/harness` is the intellectual core of the analytical engine for Engineering Insights. It acts as a compiler and execution engine, separating the declarative definition of a metric from the mechanical plumbing required to calculate and store it. It guarantees idempotency and dependency resolution for all analytical facts.

## Alignment with Product Vision

By isolating metric definitions into a pure DSL (Domain Specific Language), the harness ensures that Data Analysts or Product Managers can author new metrics (like Cycle Time or PR Size Growth) without needing to understand Step Functions, AWS Lambda constraints, or database connection pooling.

## Requirements

### Requirement 1 — The Metric DSL

**User Story:** As an analytics developer, I want a strict TypeScript contract to define metrics, so that I am forced to explicitly declare my metric's shape and dependencies without writing boilerplate.

#### Acceptance Criteria

1. The harness SHALL export a `MetricDefinition` interface and a `defineMetric` wrapper.
2. Every metric MUST explicitly declare its `tier` (`aggregate` or `analytical`), `grain` (array of valid dimensions), and `reads` (upstream dependencies).
3. Every metric MUST expose a `compute` function that returns a SQL template/query representing its raw calculations.

### Requirement 2 — The Grain Vocabulary

**User Story:** As the data layer, I want a deterministic primary key for every calculated fact, so that I can safely UPSERT data and guarantee idempotency on retries.

#### Acceptance Criteria

1. The harness SHALL restrict grain dimensions to a closed, strongly-typed vocabulary (e.g., `'repo' | 'contributor' | 'cluster' | 'pr' | 'work_item' | 'period'`).
2. The harness SHALL provide a `buildGrainKey` utility that takes an object of dimensions and returns a deterministic string.
3. The deterministic string MUST be generated by alphabetically sorting the dimension keys and joining them (e.g., `"cluster:7|period:2026-W27|repo:9"`).

### Requirement 3 — The Registry and Topological DAG

**User Story:** As the pipeline orchestration engine, I want metrics to automatically execute in the correct order, so that downstream metrics correctly read the outputs of upstream metrics.

#### Acceptance Criteria

1. The harness SHALL maintain a centralized `registry` array compiling all defined metrics.
2. The harness SHALL implement a topological sorting algorithm (e.g., Kahn's Algorithm or DFS) based on the `reads` array of each metric.
3. IF a circular dependency is detected during sorting THEN the harness SHALL throw a fatal initialization error, preventing execution.

### Requirement 4 — The Execution Adapter

**User Story:** As the Step Functions worker, I want to invoke a single class to evaluate a repository's metrics, so that I don't need to manually orchestrate the metric registry.

#### Acceptance Criteria

1. The harness SHALL export a `HarnessEngine` class that accepts a database client instance via dependency injection.
2. WHEN `runAggregation(repoId)` is called THEN the engine SHALL execute the sorted execution plan sequentially.
3. For each metric in the plan, the engine SHALL invoke the `compute` function, map the raw rows into canonical `metric_fact` shapes, generate the `grain_key`, and instruct the database client to perform a batch UPSERT.

## Non-Functional Requirements

### Code Architecture and Modularity
- **Zero Infrastructure:** The `src/registry/` folder MUST NOT contain API routing, AWS SDK calls, or database connection initialization. It strictly contains SQL definitions.
- **Type Safety:** The DSL must leverage TypeScript rigorously to prevent typos in dependency declarations or grain vocabularies.

### Performance
- The topological sort should run efficiently in-memory during class initialization, caching the execution plan for subsequent runs.
- The `HarnessEngine` MUST batch the UPSERT operations for each metric rather than performing row-by-row inserts.

### Reliability
- The deterministic sorting of the `grain_key` is a non-negotiable requirement for system idempotency. The string construction algorithm MUST be heavily unit-tested.
